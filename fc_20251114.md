# Gemini 에이전트 기반 리포트 생성 애플리케이션 분석

## 1. 프로젝트 개요

이 프로젝트는 Spring Boot를 기반으로 구축된 웹 애플리케이션으로, Google의 Gemini AI 모델을 활용하여 사용자 요청에 따라 동적으로 리포트를 생성하는 에이전트 시스템입니다. 사용자가 자연어로 질문(프롬프트)을 던지면, 시스템은 Gemini 모델의 함수 호출(Function Calling) 기능을 사용하여 필요한 데이터를 조회하고, 이를 바탕으로 최종 리포트를 생성하여 반환합니다.

## 2. 핵심 아키텍처 및 흐름

애플리케이션의 핵심은 **"Tool-Using Agent"** 패턴에 있습니다. Gemini 모델이 단순히 텍스트만 생성하는 것이 아니라, 스스로 판단하여 미리 정의된 "도구(Tool)"를 호출하고, 그 결과를 바탕으로 더 정확하고 풍부한 답변을 만들어내는 구조입니다.

### 전체 흐름

1.  **사용자 요청**: 클라이언트가 REST API 엔드포인트 (`/api/agent/report`)로 생성하고 싶은 리포트에 대한 설명(프롬프트)을 담아 POST 요청을 보냅니다.
2.  **컨트롤러 수신**: `AgentController`가 요청을 받아 `AgentService`에 처리를 위임합니다.
3.  **1차 Gemini 호출 (함수 탐색)**:
    *   `AgentService`는 `ToolRegistry`를 통해 시스템에 등록된 모든 도구(함수)의 명세(`FunctionDeclaration`)를 가져옵니다.
    *   사용자 프롬프트와 도구 명세 목록을 함께 Gemini 모델에 전달합니다.
    *   Gemini 모델은 프롬프트를 해석하고, 작업을 수행하는 데 가장 적합한 도구가 있다고 판단하면 해당 도구를 호출하라는 지시(`FunctionCall`)를 응답으로 보냅니다.
4.  **도구 실행**:
    *   `AgentService`는 Gemini의 응답에서 `FunctionCall`을 감지합니다.
    *   `ToolRegistry`를 통해 호출해야 할 도구 이름에 해당하는 `ToolExecutor` 구현체를 찾습니다.
    *   `ToolExecutor`는 Gemini가 제공한 인자(arguments)를 사용하여 실제 비즈니스 로직(예: 데이터베이스 조회)을 수행하고, 결과를 JSON 형태의 문자열로 반환합니다.
5.  **2차 Gemini 호출 (리포트 생성)**:
    *   `AgentService`는 1차 호출 때의 원본 프롬프트와 도구 실행 결과를 조합하여 새로운 프롬프트를 구성합니다.
    *   이 새로운 프롬프트와 실행 결과를 다시 Gemini 모델에 전달합니다.
    *   Gemini 모델은 전달받은 데이터를 기반으로 최종 리포트를 마크다운 형식의 텍스트로 생성합니다.
6.  **최종 응답**: `AgentService`는 Gemini가 생성한 최종 리포트 텍스트를 받아 컨트롤러를 통해 사용자에게 반환합니다.

## 3. 주요 컴포넌트 역할

### `controller`
*   **`AgentController`**: 외부 요청을 받는 유일한 진입점. `AgentService`를 호출하여 비즈니스 로직을 트리거합니다.

### `service`
*   **`AgentService`**:
    *   애플리케이션의 핵심 오케스트레이터.
    *   Gemini API와의 모든 통신(2단계 호출)을 관리합니다.
    *   `ToolRegistry`에서 도구를 조회하고, `FunctionCall`을 해석하여 적절한 `ToolExecutor`를 실행합니다.
    *   도구 실행 결과를 바탕으로 최종 리포트 생성을 Gemini에 요청합니다.
*   **`CleaningDataService`**:
    *   `CleaningReportToolExecutor`에 의해 호출됩니다.
    *   `CleaningDataRepository`를 통해 데이터베이스에서 청소 관련 데이터를 조회하는 비즈니스 로직을 수행합니다.
*   **`CompanyInfoService`**:
    *   `CompanyInfoToolExecutor`에 의해 호출됩니다.
    *   애플리케이션 시작 시, 회사 관련 정보들을 `gemini-embedding-001` 모델을 사용해 벡터 임베딩으로 변환하여 메모리에 저장합니다.
    *   사용자 질문이 들어오면, 해당 질문을 임베딩하고 코사인 유사도를 통해 가장 관련성 높은 회사 정보를 찾아 반환합니다.

### `tools`
*   **`ToolExecutor` (인터페이스)**: 시스템에 추가될 모든 도구의 규약을 정의합니다. 각 도구는 이름, 함수 명세(`FunctionDeclaration`), 실행 로직(`execute`), 프롬프트 템플릿, 시스템 지침을 제공해야 합니다.
*   **`CleaningReportToolExecutor`**:
    *   `get_cleaning_report`라는 이름의 도구를 구현합니다.
    *   `CleaningDataService`를 호출하여 특정 기간의 청소 데이터를 조회합니다.
    *   Gemini 모델이 데이터 분석가 역할을 하도록 시스템 지침("너는 데이터 분석과 리포팅 전문가야.")을 설정합니다.
*   **`CompanyInfoToolExecutor`**:
    *   `get_company_info`라는 이름의 도구를 구현합니다.
    *   `CompanyInfoService`를 호출하여 사용자 질문과 가장 유사한 회사 정보를 조회합니다.
    *   Gemini 모델이 친절한 상담가 역할을 하도록 시스템 지침("너는 친절한 상담가야.")을 설정합니다.
*   **`ToolRegistry`**:
    *   애플리케이션에 존재하는 모든 `ToolExecutor` 구현체(`@Component`로 등록된)를 자동으로 수집하고 관리하는 중앙 저장소입니다.
    *   `AgentService`가 Gemini에 전달할 도구 목록을 생성하거나, 실행할 `ToolExecutor`를 이름으로 조회할 때 사용됩니다.

### `config`
*   **`GeminiConfig`**: Gemini API 키를 주입받아 `Client` 빈을 생성합니다.
*   **`GeminiConfigFactory`**: Gemini 모델 호출 시 필요한 `GenerateContentConfig` (안전 설정, 최대 토큰 수, 도구 목록 등)를 생성하는 팩토리입니다.

### `repository` & `entity`
*   **`CleaningData`**: `CLEANING_DATA` 테이블과 매핑되는 JPA 엔티티입니다.
*   **`CleaningDataRepository`**: Spring Data JPA를 사용하여 데이터베이스와 상호작용하는 리포지토리 인터페이스입니다.

### `resources`
*   **`application.properties`**: 데이터베이스 연결 정보 및 Gemini API 키 설정이 포함됩니다.
*   **`schema.sql`, `data.sql`**: 애플리케이션 시작 시 H2 인메모리 데이터베이스의 테이블을 생성하고 초기 데이터를 삽입합니다.

## 4. 결론

이 프로젝트는 Gemini의 함수 호출 기능을 활용하여 LLM(거대 언어 모델)이 외부 시스템(데이터베이스, 벡터 검색 등)과 상호작용하는 방법을 보여주는 훌륭한 예시입니다. 도구(Tool)를 모듈식으로 설계하고 `ToolRegistry`를 통해 동적으로 관리함으로써, 새로운 기능을 손쉽게 확장할 수 있는 유연한 구조를 갖추고 있습니다.
